Anti-patterns

Things that get you paged at 2 a.m.

Anti-pattern

API_KEY = "sk-proj-abc..."

Hardcoded keys end up on GitHub. Bots scrape new commits within minutes — revocation is a race.

Anti-pattern

git add . && git commit -am "fix"

Adds .env to your commit. Even after deletion, git history keeps it forever.

Anti-pattern

print(f"Calling with key={key}")

Logs end up in CloudWatch, Datadog, Sentry, screenshots, support tickets.

Anti-pattern

sharing one key across the team

When something leaks, you can’t tell who or how. Issue keys per environment per developer.

Rule of thumb: if a stranger forked your repo right now, would they get into your account? If yes, fix it before continuing.

Gemini API

Build up from a single-line request to multimodal inputs, structured JSON, multi-turn chat, and tool calling — all on the same SDK.

Minimal request

The smallest possible Gemini request

Two required arguments
model
Which Gemini variant — flash, pro, etc.
contents
What you’re asking — string, list, or multimodal Parts.
Pick a model first
gemini-2.5-flash
Fast, cheap, default for this class.
gemini-2.5-pro
Stronger reasoning, slower, costlier.
import os
from dotenv import load_dotenv
from google import genai

load_dotenv()
client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="Explain prompt engineering in two sentences.",
)
print(response.text)
Generation parameters

Generation parameters and what they do

ParameterRangeWhat it controls
temperature 0.0 – 2.0 Randomness of next-token sampling. 0 = deterministic; ~1.5 = creative; >1.7 often degrades into nonsense.
top_p 0.0 – 1.0 Nucleus sampling: keep tokens until their cumulative probability reaches top_p. Tune one of {temp, top_p}, not both.
max_output_tokens integer Hard ceiling on response length. Set realistically — too low and you’ll see finish_reason=MAX_TOKENS truncations.
system_instruction string Persistent role / style / rules. Sent on every turn but separate from the user’s contents.
candidate_count integer Ask for multiple alternative completions in one request (model must support it).
response_schema Pydantic / dict Constrain output to a JSON shape. Pairs with response_mime_type='application/json'.
Parameters in code

Same prompt, three personalities

PROMPT = "Write one opening line for a story about a lighthouse keeper."

# Deterministic — temperature 0
boring = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=PROMPT,
    config=types.GenerateContentConfig(
        temperature=0.0,
        max_output_tokens=60,
    ),
)

# High temperature, creative
creative = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=PROMPT,
    config=types.GenerateContentConfig(temperature=1.5, top_p=0.95, max_output_tokens=60),
)

# With a system instruction (Gemini's "persona" channel — NOT a role)
styled = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=PROMPT,
    config=types.GenerateContentConfig(
        system_instruction="You are a hard-boiled noir novelist. Short, dark, cynical.",
        temperature=0.9,
        max_output_tokens=60,
    ),
)

Structured output

JSON you can trust

from pydantic import BaseModel
from typing import List

class Book(BaseModel):
    title: str
    author: str
    year: int
    genre: str

class BookList(BaseModel):
    books: List[Book]

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="List three classic sci-fi novels.",
    config=types.GenerateContentConfig(
        response_mime_type="application/json",
        response_schema=BookList,
    ),
)

# Two ways to read the result:
print(response.text)         # raw JSON string
parsed = response.parsed     # typed Python object
for b in parsed.books:
    print(b.year, b.title, "by", b.author)
Why bother
  • Predictable shape
    Your downstream code can rely on it.
  • Fewer parsing bugs
    No regex on stray markdown fences.
  • Validates types
    year is an int, not the string “nineteen sixty-nine”.
  • Production-ready
    Pair with try/except for any provider that may slip up.
Response metadata

What the response actually contains

response.text                                # convenience: first candidate's text
response.candidates                          # list of candidate completions
response.candidates[0].finish_reason         # STOP | MAX_TOKENS | SAFETY | ...
response.candidates[0].safety_ratings        # per-category classifications

response.usage_metadata.prompt_token_count
response.usage_metadata.candidates_token_count
response.usage_metadata.total_token_count    # = prompt + candidates
Always log this

usage_metadata.total_token_count is the only honest answer to “how much did this user / request / day actually cost?” — pricing pages give per-token rates.

Roles

Roles in Gemini vs. the rest of the world

ConceptGemini nameOpenAI / HF name
What the human saidrole="user""user"
What the model saidrole="model""assistant"
Persistent rules / personasystem_instruction (separate)role="system" message
Tool / function outputPart.from_function_responserole="tool"
Big trap

Gemini does NOT have a system role. Putting role="system" in your contents list will fail or behave unexpectedly. Use system_instruction in the config instead.

Multi-turn chat

Multi-turn conversations

Approach 1 — Manual history
history = [
    Content(role="user",
            parts=[Part(text="My name is Asha.")]),
]
r = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=history,
)
history.append(Content(
    role="model",
    parts=[Part(text=r.text)],
))
# next turn: append user, send again
Approach 2 — chats module
chat = client.chats.create(
    model="gemini-2.5-flash",
    config=GenerateContentConfig(
        system_instruction="Be concise.",
    ),
)

reply = chat.send_message("I'm visiting Goa.")
reply = chat.send_message("A beach for day 1?")
reply = chat.send_message("And food?")
# history is tracked for you

LLMs are stateless. Both approaches work by re-sending the full history each turn. Manual = full control; chats = less code.

Tool calling

Tool calling — six steps you’ll see in every provider

  1. Declare your functionName, description, parameter schema.
  2. Send prompt + tool listPass tools=[...] in your request config.
  3. Model decidesReplies in plain text OR returns a function_call object.
  4. You execute the functionOn your machine, with the args the model gave you.
  5. Send the result backAs a function response part in the next turn.
  6. Model finalizes the answerNow grounded on real data from your function.

Mental model: the model never executes anything. It just asks. You decide.

Tool calling in code

Tool calling — what the code looks like

def get_weather(city: str) -> dict:
    fake_db = {
        "mumbai":    {"temp_c": 31, "conditions": "humid, partly cloudy"},
        "bangalore": {"temp_c": 24, "conditions": "light rain"},
        "delhi":     {"temp_c": 36, "conditions": "hot and dry"},
    }
    return fake_db.get(city.lower(), {"error": f"No data for {city}"})

weather_tool = types.Tool(function_declarations=[
    types.FunctionDeclaration(
        name="get_weather",
        description="Get the current weather for an Indian city.",
        parameters=types.Schema(
            type=types.Type.OBJECT,
            properties={"city": types.Schema(type=types.Type.STRING)},
            required=["city"],
        ),
    ),
])

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="What's the weather in Bangalore? Should I carry an umbrella?",
    config=types.GenerateContentConfig(tools=[weather_tool]),
)

part = response.candidates[0].content.parts[0]
if part.function_call:
    fc = part.function_call
    result = get_weather(**dict(fc.args))           # 1. you run the function
    follow_up = client.models.generate_content(     # 2. send result back
        model="gemini-2.5-flash",
        contents=[..., types.Content(role="user", parts=[
            types.Part.from_function_response(name=fc.name, response=result)])],
        config=types.GenerateContentConfig(tools=[weather_tool]),
    )
    print(follow_up.text)                           # 3. final grounded answer

InferenceClient

One class, OpenAI-shaped API

The chat_completion method mirrors OpenAI’s: same messages format, same role names, same response shape. Intentional — it’s the industry’s de-facto standard. 

import os
from dotenv import load_dotenv
from huggingface_hub import InferenceClient

load_dotenv()
client = InferenceClient(token=os.environ["HF_TOKEN"])

response = client.chat_completion(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[
        {"role": "system", "content": "You are concise."},
        {"role": "user",   "content": "Explain prompt engineering in two sentences."},
    ],
    temperature=0.7,
    max_tokens=200,
)

print(response.choices[0].message.content)
print(response.usage)        # tokens, like OpenAI
print(response.choices[0].finish_reason)

Discovery + auto-routing

Find which providers host a model

from huggingface_hub import model_info

info = model_info(
    "meta-llama/Llama-3.1-8B-Instruct",
    expand=["inferenceProviderMapping"],
    token=os.environ["HF_TOKEN"],
)

When to pin vs. auto

"auto" is great for prototypes — HF picks whichever provider is fastest / cheapest / available. For production, pin a specific provider so latency and pricing stay predictable.

Structured output

JSON schema with response_format

from pydantic import BaseModel, ValidationError
from typing import List

class Book(BaseModel):
    title: str
    author: str
    year: int
    genre: str

class BookList(BaseModel):
    books: List[Book]

schema = BookList.model_json_schema()       # Pydantic gives this for free

response = client.chat_completion(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[
        {"role": "system", "content": "You return only valid JSON. No prose."},
        {"role": "user",   "content": "List three classic science fiction novels."},
    ],
    response_format={
        "type": "json_schema",
        "json_schema": {"name": "BookList", "schema": schema, "strict": True},
    },
    max_tokens=400,
)

# Defensive parsing — not every provider strictly enforces the schema
try:
    parsed = BookList.model_validate_json(response.choices[0].message.content)
except ValidationError as e:
    print("Validation failed:", e)

Loading a local model

Loading and running TinyLlama (1.1B params)

from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

MODEL_ID = "TinyLlama/TinyLlama-1.1B-Chat-v1.0"

# Pick the best device available
if torch.cuda.is_available():        device = "cuda"
elif torch.backends.mps.is_available(): device = "mps"     # Apple Silicon
else:                                device = "cpu"

# First run: pulls ~2 GB from the Hub. Cached at ~/.cache/huggingface afterwards.
tokenizer = AutoTokenizer.from_pretrained(MODEL_ID)
model = AutoModelForCausalLM.from_pretrained(
    MODEL_ID,
    torch_dtype=torch.float16 if device != "cpu" else torch.float32,
).to(device)

# Format the prompt with the model's chat template
messages = [
    {"role": "system", "content": "You are a concise assistant."},
    {"role": "user",   "content": "Explain prompt engineering in two sentences."},
]
prompt = tokenizer.apply_chat_template(
    messages, tokenize=False, add_generation_prompt=True,
)

inputs = tokenizer(prompt, return_tensors="pt").to(device)
with torch.no_grad():
    output_ids = model.generate(
        **inputs, max_new_tokens=200, do_sample=True,
        temperature=0.7, top_p=0.95,
        pad_token_id=tokenizer.eos_token_id,
    )

# Strip the prompt tokens; keep only the model's reply
new_tokens = output_ids[0][inputs["input_ids"].shape[1]:]
print(tokenizer.decode(new_tokens, skip_special_tokens=True))
Tokens & gated models

Tokens, gated models, and what “read access” means

Public

Public models

TinyLlama, Phi-2, GPT-2, etc. — no token needed for download.

Gated

Gated models

Llama, Gemma, Mistral-Instruct — accept the licence on the model page, then a read-token works.

Permissions

Read vs write

Read = pull models, run inference. Write = push models / datasets back. Use read for everything in this class.

Watch out

Storage cost

Small (~2 GB) for TinyLlama. A 70B model in fp16 is ~140 GB — read the disk size before you click download.

Tool calling

Tool calling — OpenAI shape

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get the current weather for an Indian city.",
        "parameters": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"],
        },
    },
}]

messages = [{"role": "user",
             "content": "What's the weather in Bangalore? Should I carry an umbrella?"}]

# Round 1 — model decides to call the tool
r1 = client.chat.completions.create(model="gpt-4o-mini", messages=messages, tools=tools)
assistant_msg = r1.choices[0].message
messages.append(assistant_msg)        # OpenAI requires the assistant turn in history

for call in (assistant_msg.tool_calls or []):
    args = json.loads(call.function.arguments)
    result = get_weather(**args)
    messages.append({                  # role="tool" with matching tool_call_id
        "role": "tool",
        "tool_call_id": call.id,
        "content": json.dumps(result),
    })

# Round 2 — model writes the final user-facing answer
r2 = client.chat.completions.create(model="gpt-4o-mini", messages=messages, tools=tools)
print(r2.choices[0].message.content)

Evaluating
outputs

If you can’t measure it, you can’t improve it. ROUGE for summaries, BLEU for translation, BERTScore for meaning — and humans for the rest.

ROUGE

ROUGE — n-gram overlap (summarization)

Compares candidate summary against one or more references by counting shared n-grams. Returns precision, recall, F1.

rouge1
Unigram overlap — shared single words.
rouge2
Bigram overlap — shared two-word phrases.
rougeL
Longest common subsequence — rewards in-order overlap.
from rouge_score import rouge_scorer

scorer = rouge_scorer.RougeScorer(["rouge1", "rouge2", "rougeL"], use_stemmer=True)
scores = scorer.score(reference_summary, candidate_summary)

for metric, s in scores.items():
    print(f"{metric:8} P={s.precision:.3f}  R={s.recall:.3f}  F1={s.fmeasure:.3f}")
BLEU & BERTScore

When ROUGE isn’t enough

BLEU

Translation

Precision-focused n-gram overlap with a brevity penalty.

Range: 0–100 (sacrebleu) or 0–1 (NLTK). 30+ is good for MT.

Weakness: synonyms get punished; one bad word can crush the score.

BERTScore

Semantic similarity

Embeds tokens with a transformer (RoBERTa) and matches them by cosine similarity.

Strength: recognises paraphrases — "feline rested on rug" ≈ "cat sat on mat".

Cost: slow on first run (~500 MB download); cached afterwards.

from bert_score import score

P, R, F1 = score(
    candidates, references,
    lang="en", model_type="roberta-large", verbose=False,
)
# tensors of length len(candidates) — cosine-based precision/recall/F1
End-to-end pipeline

An end-to-end comparison pipeline

# 1. Define task + gold reference
ARTICLE = "..."
PROMPT = f"Summarize the following in one sentence:\n\n{ARTICLE}"
REFERENCE = "..."         # human-written gold summary

# 2. Generate from each candidate model
outputs = {}
outputs["gemini-2.5-flash"] = call_gemini(PROMPT)
outputs["llama-3.1-8b-hf"]  = call_hf(PROMPT)

# 3. Persist (so eval is repeatable without re-spending API $$)
Path("outputs.json").write_text(json.dumps(outputs, indent=2, ensure_ascii=False))

# 4. Score against the reference
scorer = rouge_scorer.RougeScorer(["rouge1", "rouge2", "rougeL"], use_stemmer=True)
print(f"{'Model':<22} {'rouge1-F':>10} {'rouge2-F':>10} {'rougeL-F':>10}")
for label, text in outputs.items():
    s = scorer.score(REFERENCE, text)
    print(f"{label:<22} {s['rouge1'].fmeasure:>10.3f} "
          f"{s['rouge2'].fmeasure:>10.3f} {s['rougeL'].fmeasure:>10.3f}")

# 5. Read the actual outputs. Numbers lie sometimes.
for label, text in outputs.items():
    print(f"\n[{label}]\n{text}")
Limits

What automated metrics CAN’T tell you

Blind spot

Factuality

ROUGE/BLEU/BERTScore don’t know if the model invented something. A confident lie scores well if it sounds like the reference.

Blind spot

Relevance

A summary that’s correct and well-written but answers a different question still scores high if words overlap.

Blind spot

Style fit

Tone, register, formality — none of these surface in n-gram metrics.

Blind spot

Safety

Hallucinated medical advice or biased generation looks the same as good output to a metric.

Use automated metrics for fast iteration. Pair with human review and LLM-as-judge for releases.
